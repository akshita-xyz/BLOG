History of the Web

+

Understanding the history of the web helps us appreciate how far we've come and where we're heading.

Key Milestones

1989: Tim Berners-Lee proposes the World Wide Web while working at CERN

1990: First web browser and web server created

1991: World Wide Web becomes publicly available

1993: Mosaic browser released, making the web accessible to the general public

1994: Netscape Navigator released, beginning the browser wars

1995: JavaScript created, enabling dynamic web pages

1998: Google founded, revolutionizing web search

2004: Web 2.0 era begins with social media and user-generated content

2010s: Mobile-first web, responsive design, and progressive web apps

Present: AI-powered web, WebAssembly, and edge computing

How Does the Internet Work?

+

The internet is a global network of interconnected computers that communicate using standardized protocols.

Key Components

  • Servers: Computers that store and serve content
  • Clients: Devices (computers, phones) that request content
  • Routers: Direct data packets across the network
  • ISPs: Internet Service Providers that connect you to the internet
  • Protocols: Rules for communication (TCP/IP, HTTP, etc.)

How Data Travels

  1. Data is broken into small packets
  2. Each packet contains sender and destination addresses
  3. Packets travel through multiple routers to reach their destination
  4. Packets may take different routes to optimize delivery
  5. Packets are reassembled at the destination

Client-Server Architecture

+

The client-server model is the foundation of web applications, defining how clients and servers interact.

Client (Frontend)

  • User-facing interface (browser, mobile app)
  • Sends requests to the server
  • Displays data received from the server
  • Handles user interactions
  • Technologies: HTML, CSS, JavaScript, React, Vue, Angular

Server (Backend)

  • Processes requests from clients
  • Manages business logic and databases
  • Authenticates users and manages security
  • Sends responses back to clients
  • Technologies: Node.js, Python, Java, PHP, Ruby, databases

Communication Flow:

Client Request ‚Üí Server Processing ‚Üí Database Query ‚Üí Server Response ‚Üí Client Display

What Happens When You Visit a Website?

+

Understanding the journey from typing a URL to seeing a webpage is crucial for web developers.

Step-by-Step Process

1. URL Parsing

Browser breaks down the URL into protocol, domain, and path

https://www.example.com/page ‚Üí https:// + www.example.com + /page

2. DNS Lookup

Browser contacts DNS server to convert domain name to IP address

www.example.com ‚Üí 192.0.2.1

3. TCP Connection

Browser establishes TCP connection with the server (3-way handshake)

4. TLS/SSL Handshake (HTTPS)

If using HTTPS, browser and server establish secure encrypted connection

5. HTTP Request

Browser sends HTTP request to server with headers and method

6. Server Processing

Server processes the request, queries database if needed, prepares response

7. HTTP Response

Server sends HTML, CSS, JavaScript, images back to browser

8. Rendering

Browser parses HTML, applies CSS, executes JavaScript, displays page

HTTP Protocol - Explained Simply

+

HTTP (HyperText Transfer Protocol) is the language that web browsers and servers use to talk to each other. Think of it as the rules for having a conversation on the internet.

What is HTTP? (In Simple Words)

Imagine you're at a restaurant. You (the browser) look at the menu and tell the waiter (HTTP) what you want. The waiter takes your order to the kitchen (the server), and then brings back your food (the webpage). HTTP is like that waiter - it delivers your requests and brings back responses!

Key Characteristics of HTTP

üìã Request-Response Model

Your browser asks for something (request), the server gives you something back (response). It's always a two-way conversation - you ask, it answers.

Browser: "Hey, can I get the homepage?"
Server: "Sure! Here's the HTML, CSS, and images."

üîÑ Stateless Protocol

HTTP has no memory! Each request is independent and doesn't remember previous requests. It's like talking to someone with short-term memory loss - you have to introduce yourself every time.

That's why we use cookies and sessions - to help remember things like "you're logged in" or "these are items in your shopping cart."

üìù Text-Based

HTTP messages are written in plain text that humans can read (though it happens too fast for us to see). This makes it easier to debug and understand what's happening.

How HTTP Communication Works

Step 1: The Request

When you type a URL or click a link, your browser creates an HTTP request. This request contains:

  • Method: What you want to do (GET = fetch, POST = send data)
  • URL: Which page or resource you want
  • Headers: Extra information (like what browser you're using)
  • Body: Data you're sending (like form information) - only for some methods
GET /index.html HTTP/1.1
Host: www.example.com
User-Agent: Chrome/120.0
Accept: text/html
Step 2: The Response

The server processes your request and sends back an HTTP response containing:

  • Status Code: Whether it worked or not (200 = success, 404 = not found)
  • Headers: Information about the response (content type, size, etc.)
  • Body: The actual content (HTML, image, JSON data, etc.)
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 1234

<html> ... webpage content ... </html>

Common HTTP Status Codes (Response Codes)

Status codes tell you what happened with your request. They're grouped by hundreds:

2xx - Success! üéâ

  • 200 OK: Everything worked perfectly
  • 201 Created: New resource was created successfully
  • 204 No Content: Success, but nothing to send back

3xx - Redirection üîÑ

  • 301 Moved Permanently: Page moved to a new address forever
  • 302 Found: Page temporarily at a different address
  • 304 Not Modified: Use your cached version, nothing changed

4xx - Client Error (Your Mistake) ‚ùå

  • 400 Bad Request: You sent something the server doesn't understand
  • 401 Unauthorized: You need to log in first
  • 403 Forbidden: You're logged in, but you're not allowed to see this
  • 404 Not Found: The page doesn't exist (most famous error!)
  • 429 Too Many Requests: Slow down! You're making too many requests

5xx - Server Error (Server's Mistake) üí•

  • 500 Internal Server Error: Something broke on the server
  • 502 Bad Gateway: Server got an invalid response from another server
  • 503 Service Unavailable: Server is down or too busy
  • 504 Gateway Timeout: Server took too long to respond

HTTPS - The Secure Version üîí

HTTPS is just HTTP but with a security guard! The 'S' stands for Secure. It's like sending your message in a locked box instead of a postcard that anyone can read.

What Makes HTTPS Secure?

HTTPS uses encryption (SSL/TLS) to scramble data so only you and the server can understand it. Even if hackers intercept your data, they'll just see gibberish!

‚ùå HTTP (Without Encryption)

Password: mySecret123
Credit Card: 1234-5678-9012-3456

Anyone can read this!

‚úÖ HTTPS (With Encryption)

X7k2@mP9$vQ#8nL4
Fy3%dR6*hJ9&sW2!

Encrypted gibberish!

Why HTTPS Matters
  • Privacy: Nobody can spy on what you're doing
  • Security: Protects passwords, credit cards, personal information
  • Trust: Browser shows a padlock üîí, users feel safe
  • SEO: Google ranks HTTPS sites higher
  • Required: Modern browsers warn users about HTTP sites
HTTP
  • Data sent in plain text
  • Vulnerable to hackers
  • Port 80
  • No padlock in browser
  • ‚ùå Not recommended
HTTPS
  • Data encrypted (scrambled)
  • Protected from hackers
  • Port 443
  • üîí Padlock in browser
  • ‚úÖ Always use this!

Real-World Analogy üì¨

HTTP is like sending a postcard through the mail - anyone who handles it can read your message.

HTTPS is like sending a letter in a sealed, locked envelope - only you and the recipient can read it.

HTTP Methods

+

HTTP methods (also called verbs) indicate the desired action to be performed on a resource.

Method Purpose Example Use Case
GET Retrieve data from server Loading a webpage, fetching user profile
POST Send data to server to create resource Submitting a form, creating new user account
PUT Update existing resource completely Updating user profile information
PATCH Partially update existing resource Changing only the email address
DELETE Remove a resource from server Deleting a blog post or user account
HEAD Get headers without body Checking if resource exists or modified
OPTIONS Get allowed methods for resource CORS preflight requests

What is Web Hosting?

+

Web hosting is a service that stores your website files and makes them accessible on the internet.

How Hosting Works

  • Your website files are stored on a server (powerful computer)
  • Server is connected to the internet 24/7
  • When someone visits your domain, they access these files
  • Hosting provider manages server maintenance and uptime

Types of Hosting

Shared Hosting

Multiple websites on one server. Affordable but limited resources.

Good for: Small websites, blogs, personal projects

VPS (Virtual Private Server)

Dedicated portion of server resources. More control and better performance.

Good for: Growing businesses, medium-traffic sites

Dedicated Hosting

Entire server for your website. Maximum control and resources.

Good for: Large enterprises, high-traffic sites

Cloud Hosting

Resources distributed across multiple servers. Scalable and reliable.

Good for: Scalable applications, variable traffic

DNS and How It Works

+

DNS (Domain Name System) is like the phonebook of the internet, translating domain names into IP addresses.

Why DNS Exists

Computers communicate using IP addresses (like 192.0.2.1), but humans prefer memorable names (like google.com). DNS bridges this gap.

DNS Resolution Process

Step 1: Browser Cache

Browser checks its cache for the IP address

Step 2: OS Cache

If not found, checks operating system cache

Step 3: Router Cache

Queries router's cache

Step 4: ISP DNS Server

Contacts ISP's DNS resolver (recursive resolver)

Step 5: Root Name Server

ISP queries root server for TLD (.com, .org) server address

Step 6: TLD Name Server

Queries TLD server for authoritative name server

Step 7: Authoritative Name Server

Returns the actual IP address for the domain

Step 8: Response & Caching

IP address returned to browser and cached for future use

Common DNS Record Types
  • A Record: Maps domain to IPv4 address
  • AAAA Record: Maps domain to IPv6 address
  • CNAME: Canonical name, alias for another domain
  • MX Record: Mail exchange, directs email
  • TXT Record: Text information, used for verification
  • NS Record: Name server, specifies DNS servers

Browsers and How They Work

+

Web browsers are applications that retrieve, present, and traverse information on the World Wide Web.

Main Components of a Browser

  • User Interface: Address bar, back/forward buttons, bookmarks
  • Browser Engine: Marshals actions between UI and rendering engine
  • Rendering Engine: Displays requested content (HTML, CSS)
  • Networking: Handles HTTP requests and responses
  • JavaScript Engine: Parses and executes JavaScript code
  • Data Storage: Cookies, localStorage, IndexedDB

Browser Rendering Process

1. Parse HTML ‚Üí DOM Tree

Browser parses HTML and creates Document Object Model

2. Parse CSS ‚Üí CSSOM Tree

Parses CSS and creates CSS Object Model

3. Combine ‚Üí Render Tree

Combines DOM and CSSOM into render tree (only visible elements)

4. Layout (Reflow)

Calculates exact position and size of each element

5. Paint

Converts render tree to actual pixels on screen

6. Composite

Combines layers to display final page
